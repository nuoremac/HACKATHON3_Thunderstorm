PROJECT IMPLEMENTATION BLUEPRINT
Campus Radar

============================================================
1. PRODUCT VISION
============================================================

Campus Radar is a contextual connection engine for university life.

The goal is not to create another social network. The goal is to help students discover the right people, associations, events, mentors, and help opportunities at the right moment, even when profiles are incomplete.

The product solves two connected problems:

1. Students do not know who around them shares their interests, skills, availability, or academic goals.
2. Skilled students, especially quiet or highly focused students, often help others or have strong abilities, but their impact remains invisible.

Campus Radar turns weak campus data into useful connection moments and transforms student help into proof of impact.


============================================================
2. CORE PRODUCT IDEA
============================================================

The app recommends:

- students to meet;
- events to attend;
- associations to discover;
- students who can help with a skill;
- students who need help with a skill;
- study groups or project partners;
- connection moments based on timetable and availability.

Every recommendation must answer:

- Why this person?
- Why this event?
- Why now?
- What data was used?
- How confident is the system?

============================================================
5. SHOULD WE INCLUDE A CHAT?
============================================================

Recommendation: Do not build a generic social-media chat as a core feature.

Reason:
Most social media apps already have chat. If we build just another chat, the product becomes ordinary and the jury may see it as a weak social network clone.

Better approach:
Build a contextual request system first.

Instead of:
"Hello, can we talk?"

The app should generate structured requests like:

- "I need help with linear algebra."
- "I want to join a cybersecurity study group."
- "I am looking for a React teammate for a project."
- "I want feedback on my CV."
- "I want to attend this event with people from my field."

This can later open a small conversation, but the conversation must start from a clear context.

MVP decision:

- Include a lightweight message/request feature.
- Do not build a full chat system.
- Use predefined actions: Ask for help, Offer help, Join group, Attend event, Contact association.
- If time remains, add a simple message thread after a request is accepted.

Name of the feature:
Contextual Requests.

9. MATHEMATICAL MODEL
============================================================

The matching engine can use a weighted scoring model.

FinalScore(student, target) =
  w1 * InterestScore
+ w2 * SkillScore
+ w3 * AvailabilityScore
+ w4 * AcademicContextScore
+ w5 * LocationScore
+ w6 * SocialProofScore
+ w7 * FreshnessScore
+ w8 * SourceReliabilityScore
+ w9 * ExplorationScore
- p1 * MissingDataPenalty
- p2 * ConflictPenalty
- p3 * AssumptionRiskPenalty

Recommended initial weights:

- InterestScore: 0.25
- SkillScore: 0.20
- AvailabilityScore: 0.20
- AcademicContextScore: 0.10
- LocationScore: 0.10
- SocialProofScore: 0.05
- FreshnessScore: 0.03
- SourceReliabilityScore: 0.05
- ExplorationScore: 0.02

Total positive weights = 1.00

9.1 InterestScore

Measures overlap between student interests and target tags.

Formula:

InterestScore = common_tags / total_unique_tags

Example:

Student interests: AI, Robotics, Startups
Event tags: AI, Machine Learning, Projects

Common tags = 1
Total unique tags = 5
InterestScore = 1 / 5 = 0.20

9.2 SkillScore

Used for help and mentorship.

For a helper:

SkillScore =
  1.0 if exact skill match
  0.7 if related skill match
  0.3 if same field
  0.0 if unrelated

Example:

Needed skill: React
Helper skills: React, TypeScript, UI Design
SkillScore = 1.0

9.3 AvailabilityScore

Measures whether the student and target can interact at the same time.

Formula:

AvailabilityScore = overlap_minutes / required_minutes

If overlap is greater than required time, cap at 1.0.

Example:

Student free: 12:00-13:00
Event: 12:30-13:00
Required time: 30 min
Overlap: 30 min
AvailabilityScore = 1.0

9.4 AcademicContextScore

Measures academic relevance.

Suggested values:

- same department and same year: 1.0
- same department, different year: 0.8
- related department: 0.6
- different department but same interest: 0.4
- unrelated: 0.0

9.5 LocationScore

Measures physical convenience.

Suggested values:

- same building: 1.0
- nearby building: 0.7
- same campus: 0.4
- unknown location: 0.2
- far location: 0.0

9.6 SocialProofScore

Measures proof that a student, event, or association is useful.

For a helper:

SocialProofScore = positive_feedback_count / total_feedback_count

If no feedback exists:
SocialProofScore = 0.5

This avoids punishing new users too strongly.

9.7 FreshnessScore

Measures whether information is recent.

Suggested values:

- updated less than 7 days ago: 1.0
- updated less than 30 days ago: 0.7
- updated less than 90 days ago: 0.4
- older than 90 days: 0.1

9.8 SourceReliabilityScore

Suggested values:

- official association/admin source: 1.0
- imported timetable: 0.8
- student declared data: 0.7
- inferred data: 0.5
- unknown source: 0.2

9.9 MissingDataPenalty

The system must work with incomplete profiles, so missing data should reduce confidence but not block recommendations.

Suggested formula:

MissingDataPenalty = missing_important_fields / total_important_fields * 0.15

9.10 ConflictPenalty

Used when data is duplicated, contradictory, or outdated.

Examples:

- same event appears twice with different times;
- student timetable conflicts with declared availability;
- association event has no verified source.

Suggested values:

- no conflict: 0.0
- small conflict: 0.05
- medium conflict: 0.10
- major conflict: 0.20

9.11 Confidence

Confidence is different from recommendation score.

Score answers:
"How relevant is this?"

Confidence answers:
"How much do we trust the data behind this?"

Formula:

Confidence =
  0.4 * SourceReliabilityScore
+ 0.3 * ProfileCompleteness
+ 0.2 * FreshnessScore
+ 0.1 * NoConflictScore

NoConflictScore:
- 1.0 if no conflict
- 0.5 if minor conflict
- 0.0 if major conflict

9.12 ExplorationScore

This score exists because new students should not be locked inside the first assumptions the system made about them.

The app should intentionally include some recommendations that are slightly outside the student's initial profile.

Suggested values:

- target matches profile but also opens a new domain: 1.0
- target is adjacent to known interests: 0.7
- target is popular among similar students but not yet in profile: 0.5
- target is completely unrelated: 0.1

Example:

A new student selected "mathematics" and "physics".
The app can still recommend robotics, AI, entrepreneurship, or debate if there is a reasonable bridge.

9.13 AssumptionRiskPenalty

This penalty prevents a weak first assumption from becoming a destructive dependency.

Examples of risky assumptions:

- assuming a new student only wants academic events because they are in preparatory classes;
- assuming a department defines all interests;
- assuming missing profile fields mean absence of interest;
- assuming popular events are always relevant;
- assuming first-week clicks represent stable preferences.

Suggested values:

- low-risk assumption: 0.00
- medium-risk assumption: 0.05
- high-risk assumption: 0.10
- repeated unconfirmed assumption: 0.15

Implementation rule:
No single signal should dominate the recommendation score. Cap the contribution of any one signal family at 35% of the final score.

9.14 New Student Mode

For new students, the system should operate in exploration-first mode during the first weeks.

Rules:

- show recommendations from at least 3 different categories;
- mix academic, social, association, help, and career recommendations;
- ask lightweight feedback: relevant, not relevant, maybe later;
- expire unconfirmed assumptions after a short time;
- keep the explanation visible;
- let the student correct the recommendation without filling a long profile.

16. TWIST 01 RESPONSE
============================================================

Twist 01:

Apparent objective:
Suggest relevant people and events to new students.

Hidden consequence:
An initial assumption that looks neutral can become a destructive dependency if the system treats it as stable truth.

Product impact:
Campus Radar must be designed for new students without trapping them inside early assumptions.

Risk:
If a student joins the platform and the system immediately assumes that department, year, first clicks, or missing interests define the student, the app may create a filter bubble. It may keep recommending the same types of people and events, making the campus even more fragmented.

Our response:

- Treat all early data as hypotheses, not truth.
- Store source, confidence, and expiration for each signal.
- Explain which assumptions were used in every recommendation.
- Add exploration recommendations to avoid locking students into one path.
- Let users correct recommendations with one click.
- Cap the influence of any one signal family.
- Track assumption risk in the recommendation engine.

Feature added:
Assumption Ledger.

The Assumption Ledger records the assumptions behind a recommendation.

Example:

Recommendation:
"Join the Robotics discovery session."

Assumptions:
- The student may like applied science because they study mathematics and physics.
- The student is available at 14:00 according to imported timetable.
- The event is beginner-friendly according to association tags.

Risk control:
- The app asks if this is relevant.
- If ignored repeatedly, the assumption is weakened.
- If confirmed, confidence increases.

Demo line:
"For new students, Campus Radar does not pretend to know who they are. It starts with cautious hypotheses, explains them, and learns without locking the student into a fixed identity."
